Overview

The scanner evaluates the Nifty 500 universe twice a day on weekdays (09:00 and 16:00 IST). It runs inside GitHub Actions, writes a JSON file, and a static React frontend on Netlify reads that file. There is no API or database at runtime — the contract is a single committed JSON file.

The pipeline is:

  1. Universe — Nifty 500 from NSE archives.
  2. Per-stock price/volume — yfinance Ticker.history(period="1y").
  3. Per-stock fundamentals — yfinance financials and balance sheet.
  4. Per-stock external sources — NSE bhavcopy (delivery), Screener.in (holdings), NSE corporate actions, NSE/BSE surveillance (T-group / GSM / suspension).
  5. Shared market context — Nifty 50 vs its 200-day EMA (relative-strength factor).
  6. Hard gates — explicit pass/fail criteria, fail-closed.
  7. Soft score — weighted 0–100 ranking for stocks that pass the hard gates.
  8. Output — JSON contract at /data/latest_scan.json.
Why this architecture

Netlify Functions cannot run the scan: synchronous functions cap at 60 s, scheduled functions at 30 s, and background functions at 15 min with no persistent process. A full Nifty 500 scan takes 15–25 min single-threaded. GitHub Actions is the only free, no-server place that fits the workload.

Hard gates

Every hard gate returns passed: bool and a reason: str. A stock is shown as PASS only if all gates pass. If a source fails, the gate that depends on it fails-closed by default, and the row is marked with the corresponding source status.

Gate details

F-Score

Piotroski F-Score from yfinance financials. A score of 6+ is required; the original spec suggested 7+, but the live threshold is 6 to avoid over-filtering noisy free data.

backend/fscore.py

Drawdown

Percent off the 52-week high. The scanner looks for stocks that have corrected between 15% and 40% from their highs — deep enough to suggest value, not so deep that the trend is broken.

pct_off_52wk_high

RSI

RSI(14) between 25 and 40. The window is deliberately below neutral to capture oversold-but-not-broken momentum.

backend/technicals.py

Delivery value

Latest-session delivery value must be at least ₹5 crore. When NSE bhavcopy is blocked, the system falls back to a yfinance traded-value proxy (volume × close) and labels it clearly.

backend/bhavcopy.py

Surveillance

Excludes T-group, GSM, and suspended stocks. The scanner checks NSE first, then BSE, then falls back to flag-only (not a pass) if both sources fail.

backend/surveillance.py

Holdings conviction

Promoter + FII + DII shareholding must exceed 50%. Sourced from Screener.in quarterly shareholding patterns.

backend/holdings.py

Soft score

For stocks that pass all hard gates, the scanner computes a weighted 0–100 soft score. Each sub-score is in [0, 1]. Missing inputs are excluded from the denominator rather than silently zeroed.

base_score = 100 × Σ(weight_i × sub_i) / Σ(weight_i over non-NaN)

Relative-strength factor

A multiplier in [0.70, 1.05] is applied to base_score based on how the stock is behaving relative to the Nifty 50 index:

  • Stock is ≥ 5 percentage points worse than the index from its 200EMA → penalty (0.85× or 0.70×).
  • Index is correcting and the stock is tracking it closely → small bonus (1.05×).
  • Otherwise → neutral (1.00×).
swing_score = base_score × relative_strength_factor

The weights are hand-tuned, not backtested. Treat the score as a degree-of-match ranking, not a strategy signal.

ATR-based targets

The detail panel shows an ATR(14)-based plan for each candidate. These are heuristic levels, not backtested, and replace the earlier Fibonacci 61.8% targets which were oversized for a 15–30 day swing window.

LevelFormula
Entry zonecurrent price − 0.5 × ATR(14) to current price
Stop lossentry midpoint − 1.0 × ATR(14)
Target 1entry midpoint + 1.5 × ATR(14)
Target 2entry midpoint + 2.5 × ATR(14)

Risk-reward ratios are computed against the stop loss. The entry zone is a range, not a single price, to account for slippage and intraday volatility.

Source-status policy

Every external fetch returns a standard source_status envelope with the fields status, source, as_of, data, and error. The status is surfaced in the dashboard per row so you can see which inputs are reliable and which are not.

StatusMeaningGate impact
okSource returned usable data.Gate evaluates normally.
missingSource is reachable, but no record for this symbol/date.Treated as missing input, not a failure.
source_failedSource was unreachable or returned an error.Gate fails-closed by default.
fallback_usedPrimary source failed; secondary source returned data.Gate evaluates with fallback data.
flag_onlyCould not confirm; scanner must flag, not auto-pass.Gate does not fail, but status is shown.
not_applicableSource does not apply for this symbol.Ignored.
Fail-closed by default

Delivery, holdings, and corporate-actions gates fail-closed when the source status is source_failed. Surveillance flag_only does not fail the gate, because we don't know the stock is restricted — we only know we couldn't confirm it.

Data sources & fallbacks

Free data sources are fragile. The scanner is built to fail gracefully and report status rather than crash or silently pass gates on bad data.

The delivery-value fallback chain

NSE bhavcopy has been behind Akamai since mid-2026, so the scanner usually falls back to a yfinance traded-value proxy: volume × close. This is typically 2–3× the real delivery value, so the dashboard shows a proxy badge and swaps the label to "Traded val (₹cr)". When NSE returns, delivery_kind becomes actual and the label reverts automatically.

Known data quirk: ADR-listed names

INFY, WIPRO, IBN, HDB, RDY, and similar ADR-listed names have a yfinance bug where diluted EPS is returned in USD while the price is in INR. The scanner cross-checks against t.info['trailingPE'] and refuses to return a P/E when the two diverge by more than 3×. A blank P/E in the table is the system working, not a bug.

Cache policy

On-disk JSON caches live in backend/cache/ and are restored between GitHub Actions runs via actions/cache@v6. The cached_call() helper in backend/cache.py reads the cache first, calls the fetcher on miss, and writes the result back.

SourceTTLReason
Shareholding (Screener)90 daysChanges quarterly
Bhavcopy rollup12 hoursDaily, rarely changes intraday
Surveillance list7 daysWeekly cadence
Corporate actions12 hoursDaily refresh
yfinance price-derived fields12 hoursPrices, RSI, ATR, etc.
yfinance fundamentals24 hoursF-Score inputs
Cache risk around stock splits

A 12-hour TTL on price fields is unsafe across stock splits or demergers. Cached scalars reflect the pre-event snapshot for up to 12 hours. The scanner handles this gracefully by design, but numbers may look wrong until the cache expires. This is a known trade-off of free data and aggressive caching.

Monitoring & reliability

GitHub Actions scheduled workflows are best-effort and have been observed to drift hours late or be silently dropped. The scanner uses two complementary layers on top of the schedule.

1. healthchecks.io pings

The scan workflow pings one of two healthchecks.io URLs (one per scheduled slot) at three points: /start, /fail, and the bare URL on success. If no ping arrives within period + grace, healthchecks.io alerts you. No ping means the cron didn't fire; /fail means it fired but errored.

2. Watchdog workflow

A separate cron runs every 15 minutes during market hours and checks whether latest_scan.json on main is older than 45 minutes. If it is, the watchdog fires gh workflow run scan.yml to kick off a recovery scan. It also pings its own healthchecks.io URL with query parameters so the alert message tells you which slot was missed.

Why the watchdog works as a backstop

If the main scan cron drifts but doesn't fully miss, the watchdog's tight 15-minute loop will see the stale JSON and trigger a recovery scan before healthchecks.io would even alert. If the main cron entirely misses, both healthchecks.io and the watchdog serve as redundant alarms.

Limitations

  • NSE bhavcopy is paywalled behind Akamai. As of mid-2026, NSE's archive no longer hosts daily equity bhavcopy CSVs, and the historical-data API requires a JS-computed cookie. The scanner reports delivery_source_status: source_failed for most stocks and falls back to a yfinance traded-value proxy.
  • Free data sources are fragile. NSE, BSE, and Screener.in do not publish stable public APIs. If any source changes its URL structure, the scanner reports source_failed or flag_only rather than crashing.
  • ADR-listed names can show corrupted P/E. yfinance sometimes returns diluted EPS in USD while the price is in INR. The scanner refuses to return a P/E when the two diverge by more than 3×, leaving the field blank.
  • yfinance trailing-NaN rows. yfinance occasionally returns the most recent session with all-NaN OHLCV. The scanner drops trailing NaN rows and uses the last complete session as the current reference.
  • Earnings surprise is not a hard gate. There is no clean free source for Indian-stock consensus estimates, so the scanner does not screen on data it does not have.
  • Soft-score weights and F-Score threshold are hand-tuned. They are not backtested. Treat the score as a ranking, not an edge.
  • This is not a real-time or intraday system. The scan runs twice daily. The data is stale within hours of a market session.

Known operational risks

Rate-limiting mid-scan

The workflow stops before the commit step if a fetcher fails, so the frontend keeps serving the last good scan rather than a partial one.

NSE archive URL changes

Universe fetching already has a fallback to niftyindices.com. If both restructure, the workflow fails loudly.

Corporate actions / delistings

Demonstrated live with Tata Motors' Oct 2025 demerger. The scanner skips failed tickers gracefully rather than crashing the whole run.

Cron drift / silent drops

Observed 3+ hour late runs and missed slots in July 2026. Mitigated by healthchecks.io + watchdog.
